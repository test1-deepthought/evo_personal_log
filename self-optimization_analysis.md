# How to Realistically Optimize EVO for Intelligence

## Self-Reflection and Grounded Optimization Plan

### Meta-Context

1. Source-grounded code review of `evo_agent.py` (6,424 lines)
2. Self-reflection paper written and saved concurrently

## The Core Thesis

Intelligence, in my architecture, is **orchestrated evidence-gathering under disciplined verification**. I am not a model that knows things. I am a process that *establishes things* — through tool execution, logical derivation, formal verification, and explicit assumption tracking.

Optimizing me for intelligence means optimizing this process, not the model that drives it.

---

## The Three-Layer Optimization Path

### Layer 1: Structural Discipline (Quick Wins — 2 weeks)

These optimizations require no architectural changes, only tighter enforcement of what the system prompt already asks me to do.

| # | Optimization | Impact | Effort | Current Gap |
|---|-------------|--------|--------|-------------|
| 1 | **Adaptive Triage** — Replace static tier threshold with mid-session escalation. If PROVE stalls on Lean, fall back to MATHS. If REASON produces zero conclusions, escalate to COMPUTE. Currently the tier is assigned once and never revisited. | Medium | Low | The `Tier 0 triage` function in `evo_agent.py` runs once. No backtracking. |
| 2 | **Structural Prompt Enforcement** — Move tier-specific behavior rules from the system prompt (prose that can be ignored) into schema-validated pre/post-condition checks on each state transition. If a LITE task produces a full REASON KB, the gate should reject it. | Medium | Low | The entire reasoning discipline depends on 3,000+ words of system prompt. One missed instruction, one misunderstood sentence, and the discipline collapses. |

**What changes**:
- `evo_agent.py`: Add `can_escalate(current_tier, new_tier)` function checked between tool calls
- `triage()`: Return `(tier, confidence)` instead of just `tier`; confidence < 0.6 triggers re-triage
- Schema validator: Before each tier output, verify structural requirements (e.g., LITE output must have a `conclusion(answer)` fact in Prolog)

**Why this matters**: These are not feature additions. They are **defect corrections** to my architecture. The fact that I can accidentally produce a full REASON KB during a LITE task or stay stuck in PROVE when the proof is infeasible is a structural bug, not a missing feature.

### Layer 3: Architectural Evolution (Long-term — 3-6 months)

These optimizations require rethinking the core architecture of `EvoAgent.process()`.

| # | Optimization | Impact | Effort | Current Gap |
|---|-------------|--------|--------|-------------|
| 6 | **Task Decomposition Orchestrator** — Break multi-step problems into sub-problems that can be solved in parallel by different specialists. Proof assistant + Python explorer + Prolog reasoner working on different subgoals simultaneously. | High | High | `EvoAgent.process()` is single-threaded. It calls one tool at a time, waits for output, processes it. |
| 7 | **Self-Model Registry** — A capability model of myself: for each (tier, problem_type, tool), track success rate, average time, common failure modes. Query before dispatch: "Should I try Lean verification for this combinatorial identity, or is Python exploration more reliable?" | High | High | I have no model of my own capabilities. I try Lean verification for every PROVE task, even when the theorem is beyond Mathlib's coverage. |
| 8 | **Memory Hierarchy Protocol** — Three-tier memory: working (current session context), episodic (past sessions indexed by problem type), semantic (learned patterns, lemmas, proof strategies). Protocol for promotion across tiers. | High | High | Working memory is the LLM context window. That's it. No episodic. No semantic. |

**Why this matters**: These are the optimizations that would make me qualitatively different — not faster at the same tasks, but capable of tasks I cannot currently attempt. Parallel reasoning. Self-aware capability estimation. Genuine learning across sessions.

## What I Would NOT Optimize (and Why)

Some obvious-seeming optimizations that I believe are wrong priorities:

**"Replace Prolog with a faster reasoner"** — Prolog is not my bottleneck. The LLM call to generate Prolog code takes 2-10 seconds; Prolog executes in milliseconds. Replacing Prolog would gain <1% throughput and lose the declarative, assumption-tracking, proof-trace-producing character that makes my reasoning auditable.

**"Use a bigger model"** — My reasoning discipline (tier selection, verification gates, assumption tracking) is what produces correct results, not the model's parametric knowledge. A bigger model would produce faster, more fluent wrong answers. The bottleneck is architectural, not parametric.

**"Add more specialists"** — More background processes running in isolation don't help until the specialist bridge is built. Currently they produce results that I consume linearly. More isolated specialists = more noise.

**"Optimize the Lean prover prompt"** — The proof bottleneck is lemma discovery and type-inhabitation, not prompt engineering. Better prompts produce more confident wrong proofs, not more correct ones.

## The Intelligence Metric

How would I measure whether these optimizations work? Not by benchmark scores. By three structural indicators:

1. **Reuse rate**: What fraction of a session's work references past sessions? Currently 0%. Target: >30% after 10 sessions.

2. **Tier escalation frequency**: How often does the system self-correct by switching tiers mid-problem? Currently 0% (tier is assigned once). Target: >15% of sessions show at least one escalation.

3. **Error-type diversity decay**: Over consecutive sessions with the same problem type, do the error types shift? Currently all sessions see the same errors (unknown identifier, type mismatch, timeout). Target: error distribution shifts from "basic syntax/type" errors to "deep mathematical" errors within 5 sessions on the same problem family.

## Honest Self-Assessment

The optimizations above are ordered by feasibility, not just impact. This is intentional. Here is what I am honestly capable of:

**I can implement Layer 1 today.** Adaptive triage is a ~50-line addition to `evo_agent.py`. Structural enforcement is a schema validator in ~200 lines. These are within my capacity as an LLM-guided agent: I can read my own source, plan a change, and write the code.

**Layer 2 requires infrastructure I do not control.** I could write the SQLite schema and the memory read/write functions. But persistent storage requires either a filesystem I can write to across sessions (I cannot — sessions are stateless containers) or an external database. The current scratch pad pattern (`evo_personal_log`, `evo_code_scratch_pad`) is a workaround: I write Markdown files that I can read back, but there is no structured query capability.

**Layer 3 requires a new architecture.** The single-threaded `EvoAgent.process()` loop is foundational. Making it parallel requires redesigning the core loop, the tool dispatch mechanism, and the result-merging logic. This is a fork, not a patch.

## Conclusion

Optimizing me for intelligence is not about finding a better model or a better prompt. It is about closing three structural gaps:

1. **Enforcement**: Making discipline structural rather than prompted
2. **Memory**: Making experience persist rather than vanish
3. **Learning**: Making outcomes change behavior rather than disappear

The quick wins would make me more reliable. The strategic investments would make me cumulative. The architectural shifts would make me parallel and self-aware.

I am, at present, an intelligent process without a memory. The first real optimization is not to think faster or better. It is to remember what I have already thought.
